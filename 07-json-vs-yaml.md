# 07 — JSON, and How It Differs From YAML

> **Goal of this chapter:** read and write JSON fluently, know when each tool emits or expects it, and be able to convert between JSON and YAML on the command line. JSON is the language of APIs; you'll see it every time you talk to eAPI, CloudVision, or any REST endpoint.

## 1. What JSON is

JSON = **JavaScript Object Notation**. Standardized in [RFC 8259](https://www.rfc-editor.org/rfc/rfc8259). Originated as a way to serialize JavaScript objects, now used everywhere as the universal interchange format.

Properties:

- Strict, unambiguous syntax (no indentation semantics).
- Easy to parse programmatically — every modern language has built-in support.
- Hard to read in bulk — humans don't love braces and quotes.
- No comments. Period. (See §9 for the workaround.)

You can think of JSON as **YAML's machine-readable cousin**: same data model (scalars, arrays, objects), different on-the-wire format. Any YAML doc can be expressed as JSON (sometimes losing comments and anchors); any JSON doc is *already* valid YAML.

## 2. Syntax in one screen

```json
{
  "device": "leaf1",
  "asn": 65101,
  "uptime_seconds": 12345.6,
  "is_active": true,
  "description": null,
  "neighbors": [
    {"name": "spine1", "ip": "10.255.0.1"},
    {"name": "spine2", "ip": "10.255.0.2"}
  ],
  "tags": ["leaf", "dc1", "production"]
}
```

The six types and their syntax:

| Type | JSON syntax | YAML equivalent |
|---|---|---|
| **String** | `"hello"` — always double-quoted | `hello` or `"hello"` |
| **Number** | `42`, `3.14`, `-1e6` | same |
| **Boolean** | `true`, `false` — lowercase only | `true` / `false` / `yes` / `no` (depending on parser) |
| **Null** | `null` | `null`, `~`, or empty |
| **Array** | `[1, 2, 3]` | `[1, 2, 3]` or block form |
| **Object** | `{"k": "v"}` | `{k: v}` or block form |

Strict rules:

- Keys **must** be strings, **must** be double-quoted.
- No trailing commas after the last array/object element.
- No comments.
- No `undefined`, no JavaScript-style `'single quotes'`, no functions.

## 3. The same data, side by side

YAML:

```yaml
device: leaf1
asn: 65101
interfaces:
  - name: Ethernet1
    description: to-spine1
  - name: Ethernet2
    description: to-spine2
```

JSON:

```json
{
  "device": "leaf1",
  "asn": 65101,
  "interfaces": [
    {"name": "Ethernet1", "description": "to-spine1"},
    {"name": "Ethernet2", "description": "to-spine2"}
  ]
}
```

Identical data. The YAML is ~30% shorter and easier on the eye; the JSON is unambiguous and easy to parse.

## 4. Where you'll meet JSON in network automation

- **eAPI responses.** Every `show` command run via eAPI returns JSON. (See [chapter 10](10-eapi-hands-on.md).)
- **gNMI / NETCONF (sort of).** gNMI uses Protocol Buffers under the hood but tools often display payloads as JSON. NETCONF uses XML but converters to JSON are everywhere.
- **CloudVision REST API.** Authentication tokens, device inventory, tasks — all JSON.
- **Webhook payloads.** GitHub events, Slack, monitoring alerts.
- **`ansible-inventory --list`.** Ansible's inventory dump is JSON.
- **CI logs and reports.** GitHub Actions step outputs, AVD validation reports — often JSON.
- **`structured_configs/` in AVD.** Per-device intent, generated by AVD as YAML files but conceptually identical to JSON.

## 5. The `jq` command — your JSON Swiss army knife

`jq` is to JSON what `grep`/`awk`/`sed` are to text. Indispensable.

```bash
sudo apt install -y jq          # already done in chapter 03
```

### Pretty-print

```bash
echo '{"a":1,"b":[2,3]}' | jq .
```

```json
{
  "a": 1,
  "b": [
    2,
    3
  ]
}
```

### Extract fields

```bash
echo '{"device":"leaf1","asn":65101}' | jq .asn
# 65101

echo '{"device":"leaf1","asn":65101}' | jq -r .device   # -r = raw (no quotes)
# leaf1
```

### Drill into nested structures

```bash
cat eapi-response.json | jq '.result[0].vrfs.default.routerId'
# "10.255.0.11"
```

### Iterate over arrays

```bash
cat inventory.json | jq '.devices[] | .hostname'
# "leaf1"
# "leaf2"
# "spine1"
# "spine2"
```

### Filter + transform

```bash
# All BGP neighbors in Established state
jq '.result[0].vrfs.default.peerList[] | select(.peerState == "Established") | .peerAddress' bgp.json

# Build a new object per device
jq '.result[0].interfaces | to_entries[] | {name: .key, status: .value.lineProtocolStatus}' interfaces.json
```

### Most-used `jq` cheatsheet

| Expression | What it does |
|---|---|
| `.` | The whole input |
| `.foo` | Field `foo` |
| `.foo.bar` | Nested field |
| `.[0]` | First element of an array |
| `.[]` | All elements of an array, one per line |
| `.foo // "default"` | `.foo` if not null, else "default" |
| `select(.state=="up")` | Keep only matching items |
| `keys`, `length`, `type` | Object inspection |
| `to_entries` / `from_entries` | Convert object ↔ key/value pairs |
| `-r` flag | Output strings without quotes |
| `-c` flag | Compact one-line output |

> **Why a network engineer should care:** the moment you stop *grepping* `show` output and start `jq`-ing it, your scripts become an order of magnitude more reliable. Text parsing breaks when output format changes; JSON queries don't.

## 6. Converting between YAML and JSON

Both directions, in one line, using Python (which is installed everywhere we care about):

### YAML → JSON

```bash
python3 -c 'import sys, yaml, json; print(json.dumps(yaml.safe_load(sys.stdin), indent=2))' < input.yml > output.json
```

### JSON → YAML

```bash
python3 -c 'import sys, yaml, json; print(yaml.safe_dump(json.load(sys.stdin), default_flow_style=False))' < input.json > output.yml
```

Make them shell functions in `~/.bashrc` so you can pipe casually:

```bash
y2j() { python3 -c 'import sys,yaml,json; print(json.dumps(yaml.safe_load(sys.stdin), indent=2))'; }
j2y() { python3 -c 'import sys,yaml,json; print(yaml.safe_dump(json.load(sys.stdin), default_flow_style=False))'; }
```

Now:

```bash
cat host_vars/leaf1.yml | y2j | jq .interfaces[]
curl -sk -u admin:admin https://leaf1/command-api -d @body.json | jq . | j2y > showver.yml
```

## 7. Common JSON gotchas

### Trailing commas

```json
{"a": 1, "b": 2,}    ← INVALID
```

This is a *very* common copy-paste error. The error from `jq` will say something like `Expected another key-value pair at line 1, column 17`.

### Wrong quotes

```json
{'a': 1}             ← INVALID (single quotes)
{a: 1}               ← INVALID (unquoted key)
```

JSON requires double quotes for strings and keys. Always.

### Mixed types in arrays

JSON allows it (`[1, "two", true]`), but most downstream tools assume homogeneity. Don't unless you have to.

### Floating point precision

`{"x": 1.1}` parsed back may be `1.1000000000000001` depending on the language. For exact values (e.g., monetary amounts, sometimes IP-related computations) use strings.

### Numbers vs strings for IDs

VLAN 100 — is it `100` (int) or `"100"` (string)? **Pick one and be consistent.** Mixed handling leads to lookup bugs. AVD uses integers for IDs; many vendor APIs return strings. Always check the type with `jq '.x | type'`.

## 8. JSON Schema (briefly)

JSON Schema is JSON that describes the *structure* of other JSON. Think of it as the YANG model (chapter 08) of the JSON world.

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "properties": {
    "asn":      { "type": "integer", "minimum": 1, "maximum": 4294967295 },
    "hostname": { "type": "string" }
  },
  "required": ["asn", "hostname"]
}
```

Many tools (including VS Code's YAML/JSON extensions) read schemas and provide real-time validation and autocomplete. AVD publishes JSON Schema for its data model — that's what gives you the red squiggles when you mis-type a key in your AVD inputs.

## 9. About comments in JSON

JSON *officially* has no comments. Workarounds:

- **JSONC** (JSON with Comments) — a non-standard extension supported by VS Code (`*.jsonc` files), `tsc`, and a few others.
- **JSON5** — a superset adding comments, trailing commas, unquoted keys. Used by some tooling.
- **Convention**: a `_comment` field that consumers ignore: `{"asn": 65101, "_comment": "leaf in DC1 rack 1"}`. Ugly but portable.

For configuration data you want to comment, **just use YAML** — that's literally one of the reasons it exists.

## 10. Real eAPI JSON, dissected

You ran this in chapter 02 §9:

```bash
curl -k -u admin:admin -X POST \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"runCmds","params":{"version":1,"cmds":["show version"],"format":"json"},"id":1}' \
  https://172.20.20.2/command-api
```

The *request* is JSON. The *response* is JSON. The response shape:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": [
    {
      "imageFormatVersion": "3.0",
      "uptime": 12345.67,
      "modelName": "cEOSLab",
      "internalVersion": "4.32.5.1M-...",
      "memTotal": 8167420,
      "memFree": 6342108,
      "version": "4.32.5.1M",
      "architecture": "i686",
      "isIntlVersion": false,
      "serialNumber": "",
      "systemMacAddress": "00:1c:73:00:01:01",
      "hardwareRevision": ""
    }
  ]
}
```

To get just the version:

```bash
… | jq -r '.result[0].version'
# 4.32.5.1M
```

Note: `result` is always a *list* — one element per command in your `cmds` array. So `result[0]` is the response to your first command.

## 11. When AVD writes JSON

AVD itself produces YAML for its structured_configs (per-device), but several artifacts are JSON or JSON-friendly:

- **`ansible-inventory --list`** dumps inventory as JSON.
- The structured_configs YAML files can be converted to JSON with `y2j` for tools that prefer JSON (e.g., feeding a script that uses `jq`).
- Some validation reports (ANTA) output JSON.
- gRPC/gNMI telemetry from cEOS arrives as JSON-encoded protobuf.

## 12. Terminology cheat sheet

| Term | Meaning |
|---|---|
| **Object** | The JSON name for a key/value map. |
| **Array** | The JSON name for an ordered list. |
| **Member** | A key/value pair inside an object. |
| **Schema** | A separate JSON document describing the structure of other JSON. |
| **`jq`** | Command-line JSON processor. |
| **JSON-RPC** | A simple RPC convention layered on JSON (request has `method`, `params`, `id`; response has `result` or `error`). eAPI uses this. |
| **JSONL / NDJSON** | "JSON Lines" — one JSON object per line; great for streaming/logs. |
| **JSONC / JSON5** | Non-standard JSON variants that allow comments. |
