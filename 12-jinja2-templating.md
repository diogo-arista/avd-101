# 12 — Jinja2 Templating

> **Goal of this chapter:** read and write Jinja2 templates with confidence. By the end you'll be able to render any data structure into device configuration, write conditional logic, use loops and filters, and understand AVD's `eos_cli_config_gen` templates when you read them.

We introduced Jinja briefly in [chapter 00 §L5](00-network-automation-101.md) and again in [chapter 01 §3](01-avd-guided-tour.md). This is the deep dive.

## 1. The mental model

Jinja takes two inputs:

```
┌─────────────────┐     ┌─────────────────┐
│   TEMPLATE      │  +  │     DATA        │  →  rendered text
│  (text + tags)  │     │  (dict / list)  │
└─────────────────┘     └─────────────────┘
```

Templates are text files (any text — Cisco config, EOS config, HTML, YAML, anything) with **placeholders**. When rendered, the placeholders are replaced with values from the data.

The "data" in network-automation contexts is usually a Python dict (often loaded from YAML).

## 2. The three syntax markers

| Marker | Purpose | Example |
|---|---|---|
| `{{ expression }}` | Print the value | `vlan {{ id }}` |
| `{% statement %}` | Control flow (no output) | `{% for v in vlans %}...{% endfor %}` |
| `{# comment #}` | Removed at render time | `{# generated by AVD #}` |

The dash variants `{{- ... -}}`, `{%- ... -%}`, `{#- ... -#}` strip surrounding whitespace — handy when you don't want blank lines polluting output.

## 3. Your first template

Make a directory `labs/12-jinja/` and create `data.yml`:

```yaml
hostname: leaf1
vlans:
  - id: 100
    name: USERS
  - id: 200
    name: VOICE
  - id: 300
    name: GUEST
```

Create `template.j2`:

```jinja
hostname {{ hostname }}
!
{% for v in vlans %}
vlan {{ v.id }}
   name {{ v.name }}
!
{% endfor %}
```

Render it:

```python
# render.py
import yaml
from jinja2 import Template
from pathlib import Path

data = yaml.safe_load(Path("data.yml").read_text())
tpl = Template(Path("template.j2").read_text())
print(tpl.render(**data))
```

```bash
python3 render.py
```

Output:

```
hostname leaf1
!
vlan 100
   name USERS
!
vlan 200
   name VOICE
!
vlan 300
   name GUEST
!
```

Add a fourth VLAN to `data.yml`, rerun — a fourth stanza appears. One template, any number of VLANs.

## 4. Variables and attribute access

```jinja
{{ vlan.id }}            {# attribute access via dot #}
{{ vlan['id'] }}         {# subscript access (equivalent) #}
{{ device.interfaces[0].name }}    {# nested + index #}
{{ vars.fabric.asn_range }}        {# arbitrarily deep #}
```

Both `.attr` and `['attr']` work for dicts. The dot form is preferred for readability; use brackets when the key has weird characters or comes from a variable.

## 5. Filters: pipe-style transforms

Filters are functions you apply with `|`:

```jinja
{{ name | upper }}                {# "leaf1" → "LEAF1" #}
{{ asn | string }}                {# int → str #}
{{ count | default(0) }}          {# fallback if undefined #}
{{ items | length }}              {# count #}
{{ items | join(', ') }}          {# list to string #}
{{ name | replace('-', '_') }}    {# string replacement #}
{{ value | trim }}                {# strip whitespace #}
{{ asn | int }}                   {# cast to int #}
```

Chain them:

```jinja
{{ description | lower | replace(' ', '_') }}
```

### Networking-flavored filters (from Ansible / AVD)

```jinja
{{ '10.0.0.1/24' | ipaddr('network') }}     {# 10.0.0.0 #}
{{ '10.0.0.1/24' | ipaddr('prefix') }}      {# 24 #}
{{ vlans | arista.avd.natural_sort('id') }} {# sort by id, alphanum-aware #}
{{ value is arista.avd.defined }}           {# AVD's "defined and not None" test #}
```

Most "networking" filters come from Ansible (`ansible.utils.ipaddr`, `community.general.netaddr`) or AVD's own filter library (`arista.avd.natural_sort`, `arista.avd.range_expand`, etc.).

## 6. Conditionals

```jinja
{% if interface.lacp %}
   lacp port-priority 100
{% elif interface.spanning_tree %}
   spanning-tree portfast
{% else %}
   {# nothing #}
{% endif %}
```

Tests with `is`:

```jinja
{% if name is defined %}              {# variable is set (even if None or empty) #}
{% if name is none %}                  {# variable is None #}
{% if items is iterable %}             {# can loop over #}
{% if items is sequence %}             {# is a list/tuple #}
{% if x is divisibleby(2) %}
```

In AVD code you'll see `is arista.avd.defined` — a stricter test that also rules out `None`. It's the safe pattern for AVD inputs since users may leave keys absent vs explicitly null.

## 7. Loops in depth

### Basics

```jinja
{% for vlan in vlans %}
vlan {{ vlan.id }}
{% endfor %}
```

### Loop variables

Inside a `{% for %}`, you get a `loop` object:

```jinja
{% for v in vlans %}
vlan {{ v.id }}   {# {{ loop.index }} of {{ loop.length }} #}
{% if not loop.last %}
!
{% endif %}
{% endfor %}
```

| `loop.*` | Meaning |
|---|---|
| `loop.index` | 1-based counter |
| `loop.index0` | 0-based counter |
| `loop.first` | True for first iter |
| `loop.last` | True for last iter |
| `loop.length` | total iterations |
| `loop.revindex` | counts down |

### Iterating dicts

```jinja
{% for name, port in interfaces.items() %}
interface {{ name }}
   description {{ port.description }}
{% endfor %}
```

### Conditional loops

```jinja
{% for v in vlans if v.id < 1000 %}
vlan {{ v.id }}
{% endfor %}
```

### Sorting + filtering

```jinja
{% for n in neighbors | sort(attribute='ip') %}
neighbor {{ n.ip }} description {{ n.desc }}
{% endfor %}

{% for v in vlans | selectattr('tenant', 'equalto', 'TENANT_A') %}
vlan {{ v.id }} {# only TENANT_A vlans #}
{% endfor %}
```

## 8. Whitespace control (the most painful part)

A naive template:

```jinja
{% for v in vlans %}
vlan {{ v.id }}
{% endfor %}
```

…renders with extra blank lines between blocks because the `{% for %}` and `{% endfor %}` *occupy lines* in the source. The fix: dash variants that strip whitespace.

```jinja
{% for v in vlans -%}
vlan {{ v.id }}
{% endfor -%}
```

| Marker | What it strips |
|---|---|
| `{%-` | Whitespace *before* the tag |
| `-%}` | Whitespace *after* the tag |
| `{{-` / `-}}` | Same for expressions |

Mastering whitespace control is what separates "this template kind of works" from "this template produces config indistinguishable from what a human would write." AVD's templates are extreme about whitespace — open any of them and count the dashes.

You can also configure the Jinja environment globally:

```python
env = Environment(
    trim_blocks=True,        # strip newline after a block tag
    lstrip_blocks=True,      # strip leading whitespace before a block tag
    keep_trailing_newline=True,
)
```

`trim_blocks + lstrip_blocks` are nearly always what you want for config generation.

## 9. Macros: reusable blocks

```jinja
{% macro interface_stanza(name, descr=None, mtu=9214) %}
interface {{ name }}
{% if descr %}
   description {{ descr }}
{% endif %}
   mtu {{ mtu }}
!
{% endmacro %}

{{ interface_stanza('Ethernet1', descr='to-spine1') }}
{{ interface_stanza('Ethernet2', descr='to-spine2', mtu=1500) }}
```

Macros are functions for templates. AVD uses them extensively — each "feature template" is a set of macros invoked by a parent template.

## 10. Includes and inheritance

### Include another template inline

```jinja
{% include 'banner.j2' %}
```

The included file is rendered with the *current* template's variables.

### Template inheritance (`extends`)

A "base" template defines blocks; a child template overrides them.

```jinja
{# base.j2 #}
{% block header %}!{% endblock %}
{% block interfaces %}{% endblock %}
{% block footer %}end{% endblock %}
```

```jinja
{# leaf.j2 #}
{% extends 'base.j2' %}
{% block header %}
hostname {{ hostname }}
!
{% endblock %}
{% block interfaces %}
{% include 'interfaces.j2' %}
{% endblock %}
```

This is how AVD's `eos_cli_config_gen` orchestrates: one top-level template `eos-intended-config.j2` includes feature templates per section.

## 11. Custom filters and tests (Python side)

When you need logic that's awkward in Jinja, write a Python filter and register it:

```python
from jinja2 import Environment

def mac_format(mac, sep=':'):
    """Normalize MAC like 'aabb.ccdd.eeff' to 'aa:bb:cc:dd:ee:ff'."""
    digits = ''.join(c for c in mac if c.isalnum())
    return sep.join(digits[i:i+2] for i in range(0, 12, 2))

env = Environment()
env.filters['mac_format'] = mac_format

tpl = env.from_string("mac {{ '0011.2233.4455' | mac_format }}")
print(tpl.render())
# mac 00:11:22:33:44:55
```

Ansible plugins put this in `filter_plugins/myfilters.py` as a class. AVD ships dozens of custom filters that way.

## 12. Debugging templates

- **`{{ var | type_debug }}`** in Ansible — prints the type.
- **Render in isolation**: `python3 -c "from jinja2 import Template; print(Template(open('x.j2').read()).render(...))"`.
- **`ansible-playbook --check --diff`** shows what a template *would* produce.
- For AVD: examine the structured_config YAML (one level above the templates) to confirm the data is right before blaming the template.

⚠ The two most common errors:
- **`UndefinedError`** — you referenced a key the data doesn't have. Use `default(...)` or guard with `if x is defined`.
- **Silent wrong output** — whitespace, off-by-one indentation, or a missing colon. Render then `diff` against a known-good config.

## 13. Where Jinja shows up in AVD specifically

- **`eos_cli_config_gen/templates/**/*.j2`** — the templates that render structured config to EOS CLI. Hundreds of them, organized per feature (BGP, MLAG, ACLs, etc.).
- **Inside Ansible playbooks** — task names, variable expressions, conditionals all use Jinja.
- **`group_vars` / `host_vars`** — values can be Jinja expressions referencing other vars: `mlag_peer_address: "{{ mlag_peer_loopback }}/{{ mlag_peer_mask }}"`.
- **AVD documentation templates** — the per-device Markdown docs are also rendered with Jinja.

When you eventually open an AVD template, expect:
- Heavy use of `arista.avd.natural_sort` and `arista.avd.defined`.
- Whitespace-stripping dashes everywhere.
- Nested includes and macros.
- Comments that label sections.

It looks intimidating but the building blocks are all in this chapter.

## 14. Terminology cheat sheet

| Term | Meaning |
|---|---|
| **Template** | A text file with Jinja markers. |
| **Render** | The process of turning a template + data into final text. |
| **Context** | The variables available to the template at render time. |
| **Filter** | A `|`-applied transform: `{{ x | upper }}`. |
| **Test** | An `is`-applied check: `{% if x is defined %}`. |
| **Macro** | A reusable block (like a function). |
| **Block** | A named section in a base template that children can override. |
| **Strict mode** | Jinja config where referencing an undefined variable raises (Ansible's `strict_concurrency` does this too). |
| **Autoescape** | HTML-safety feature; we leave it off for config generation. |
